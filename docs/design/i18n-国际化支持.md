# 技术方案: i18n 国际化支持

**日期:** 2026-06-29
**状态:** Draft

## 1. 需求概述

当前项目所有 UI 文本（约 80+ 条）均为硬编码中文，无法适配英文用户。项目已有 `UserPreferences.language` 字段（`'zh-CN' | 'en'`），但缺乏 i18n 基础设施。

**目标：** 基于 React Context + JSON 语言包的自建方案，实现全界面中/英文切换，无需新增第三方依赖。

**成功标准：**
- 所有 UI 文本可切换中/英文
- 日期、时间、数字格式化跟随当前语言
- 语言切换即时生效（通过 React Context 传播）
- 首次启动自动检测浏览器语言
- 新增翻译只需修改 JSON，不改组件逻辑

## 2. 技术栈

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| i18n 框架 | React Context + JSON（自建） | PRD 要求不引入第三方库，控制包体积 < 5KB |
| 语言包 | TypeScript 静态 import JSON | 避免动态加载带来的异步延迟，利用构建时 tree-shaking |
| 状态管理 | ConfigStore（已有） | 复用 `preferences.language` 字段，通过 `onChange` 监听跨标签页同步 |
| 格式化 | `Intl` 原生 API | `toLocaleString` / `toLocaleTimeString` 已内置，零依赖 |

### 3.2 模块划分

| 模块 | 职责 | 输入 | 输出 | 依赖 |
|------|------|------|------|------|
| `i18n/types.ts` | 类型定义：`Locale`、`MessageSchema`、`I18nContextValue` | - | 类型导出 | - |
| `i18n/language-detector.ts` | 首次语言检测：`navigator.language` 匹配逻辑 | 无 | `'zh-CN' \| 'en'` | `ConfigStore` |
| `i18n/I18nProvider.tsx` | Context Provider：加载语言包、监听配置变更 | `children: ReactNode` | `<I18nContext.Provider>` | `ConfigStore`、语言包 JSON |
| `i18n/useI18n.ts` | useI18n hook：返回 `t()`、`locale`、`setLanguage()` | - | `I18nContextValue` | `I18nProvider` |
| `locales/zh-CN.json` | 中文语言包 | - | 静态 JSON | - |
| `locales/en.json` | 英文语言包 | - | 静态 JSON | - |
| `utils.ts` (改造) | `formatTime(locale)`、`formatDateTime(locale)`、`formatNum(n, locale)` 增加 locale 参数 | `locale: string` | 格式化字符串 | - |
| 13 个组件 (改造) | 用 `t('key')` 替换硬编码中文文本 | `useI18n()` | JSX | `useI18n` |

### 3.3 数据流向

```
1. 首次启动:
   navigator.language → 检测匹配 → ConfigStore.set('preferences', {..., language: 'zh-CN'|'en'})

2. 渲染流程:
   ConfigStore.get('preferences').language → I18nProvider state
   → 静态 import 对应 JSON → messages 对象
   → 注入 Context (t, locale, setLanguage)
   → 各组件 useI18n() 消费

3. 语言切换:
   用户在设置面板选择语言 → setLanguage('en')
   → ConfigStore.set('preferences', {..., language: 'en'})
   → ConfigStore.onChange 触发 I18nProvider 更新
   → Context value 变更 → 所有消费者重渲染

4. 跨标签页同步:
   chrome.storage.onChanged 事件 → ConfigStore.onChange
   → I18nProvider 监听到 preferences 变更 → 自动更新 language
```

### 4.5 t() 函数实现规范

```typescript
/**
 * 模板变量语法：使用双大括号 {{varName}}
 * 示例：
 *   t('token.total', { total: 1234 }) → "总计 1,234"
 *   t('voice.syncComplete', { count: 5 }) → "同步完成，共 5 个技能"
 */
function resolveTemplate(template: string, vars?: Record<string, string | number>): string {
  if (!vars) return template;
  return template.replace(/\{\{(\w+)\}\}/g, (_, key) => String(vars[key] ?? `{{${key}}}`));
}
```

Key 为点分隔路径，如 `'sidebar.status.running'`，`t()` 内部按路径从 messages 对象取值，再应用模板变量替换。

### markdown-viewer（独立入口）处理

`markdown-viewer` 是独立 HTML 页面，不在 React 树中。方案：

```typescript
// 从 markdown-viewer/index.ts 改造

// 1. 从 chrome.storage.local 读取 preferences.language
// 2. 动态 import 对应语言包 (zh-CN.json 或 en.json)
// 3. 使用 messages.markdown.xxx 替换硬编码文本
// 4. 由于这是独立 entry，不通过 React Context，直接用函数式获取

async function getMarkdownMessages(): Promise<MessageSchema['markdown']> {
  const { preferences } = await browser.storage.local.get('preferences');
  const lang = preferences?.language ?? 'zh-CN';
  if (lang === 'en') {
    return (await import('../sidepanel/locales/en.json')).markdown;
  }
  return (await import('../sidepanel/locales/zh-CN.json')).markdown;
}
```

### 风险点及对策

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 语言包 key 遗漏 | 某些文本仍显示中文/缺失 | 高 | 使用 TypeScript 类型定义确保 MessageSchema 覆盖所有文本；写一个运行时 fallback 逻辑：`t()` 若 key 不存在则返回 key 本身并 console.warn |
| 模板变量命名不一致 | 翻译后显示原始 `{{var}}` 占位符 | 中 | 统一文档规范，t() 函数内对未替换的占位符打印 warning |
| 跨标签页语言同步延迟 | 已打开的 sidepanel 不及时响应 | 低 | ConfigStore.onChange 已支持 chrome.storage.onChanged 监听，I18nProvider 订阅即可 |
| markdown-viewer 语言包重复引用 | 包体积增大 | 低 | Vite 的 chunk 共享机制自动去重，同源 JSON import 只保留一份 |
| 日期格式化因 locale 变化导致 SSR/CSR 不一致 | 测试环境差异 | 低 | 纯客户端渲染，无 SSR，不涉及 |

## 7. 关键决策

| 决策 | 方案 | 备选 | 理由 |
|------|------|------|------|
| 框架选型 | 自建 React Context | react-i18next / i18next | PRD 要求不引入第三方库，自建方案 < 5KB |
| 语言包加载方式 | 静态 import JSON | 动态 import() 或 fetch | 静态 import 零延迟、类型安全、tree-shaking 友好 |
| 模板变量语法 | `{{varName}}` 双大括号 | `{varName}` 或 `%{varName}` | 与主流 i18n 库一致，不会与 JSX `{}` 混淆 |
| 语言检测时机 | `main.tsx` 渲染前同步执行 | useEffect 中异步执行 | 避免首次渲染时语言未就绪导致的闪烁 |
| 数字格式化 | 原生 `toLocaleString(locale)` | 自建格式化 | 零依赖、覆盖所有场景、浏览器内置 |
| markdown-viewer | 独立读取 storage + import JSON | 注入到 React 树 | 该入口是独立 HTML 页，无法复用 React Context |
| 语言选择器位置 | 设置面板 Provider Tab 下方 | 独立全局设置入口 | 符合 PRD 要求，与 Provider 配置同一层级 |

## 8. 非功能需求实现方案

### 8.1 性能
- 语言包通过静态 import 在构建时内联，无运行时网络请求
- 语言切换仅触发 Context 更新，React 自动批量重渲染，无额外开销
- 两个语言包各约 3-5KB gzip 前，构建后增量 < 8KB（满足 < 5KB gzip 要求）
- `t()` 函数使用简单的路径解析 + 正则替换，O(路径深度 + 模板变量数)，< 0.1ms

### 8.2 安全
- 语言包为纯 JSON（静态 import 编译时校验），不含可执行逻辑
- 模板变量通过 `String()` 安全转义，禁止 HTML 注入
- 错误消息中动态内容使用 `sanitizeText()`（已有）二次确认

### 8.3 可维护性
- 新增翻译仅需修改两个 JSON 文件，不改组件代码
- `MessageSchema` 类型为所有 key 提供编译时检查
- Key 按组件命名空间组织：`sidebar.*`、`settings.provider.*`，直观可定位
- 新增语言仅需添加一个新的 JSON 文件 + 扩展 `Locale` 类型

### 8.4 向后兼容
- 所有组件 Props 保持不变
- `formatTime`/`formatDateTime` 增加 locale 参数但默认值为 `'zh-CN'`，不传时行为不变
- `config-store.ts` 的 `DEFAULTS.preferences.language` 保持 `'zh-CN'`
