# PRD: 优化聊天首次体验与响应式布局

**日期:** 2026-07-18
**状态:** Confirmed
**优先级:** P0

## 1. 概述

### 1.1 问题陈述

当前聊天首页与已有对话页使用不同的宽度约束：首页主内容接近铺满，而聊天消息流和底部输入框受固定最大宽度限制，导致两种状态切换时视觉差异明显；在 1080P、2K 等桌面视口下，聊天内容区又显得过窄。

消息输入框内的 Provider、Model、Reasoning 使用浏览器原生下拉弹层，无法与现有主题保持一致。首次进入且尚未配置 Provider 时，输入被禁用，但页面没有明确说明下一步，也没有主动引导用户完成配置。

### 1.2 目标用户

- 首次安装并打开插件、尚未配置 Provider 的用户
- 已使用插件，但当前不存在完整 Provider 配置的用户
- 在不同桌面视口宽度下使用聊天功能的现有用户

### 1.3 业务价值

- 降低首次使用门槛，让用户明确完成 Provider 配置后再开始聊天
- 消除首页与聊天页之间的布局跳变
- 提高桌面宽屏空间利用率和输入配置操作的一致性
- 避免用户在无 Provider 时面对不可操作但没有解释的输入框

### 1.4 成功标准

- 首页与聊天页所有主内容在同一视口下具有一致的左右边界
- 可用视口宽度为 `1024–1439px` 时，主内容宽度为 90%；`≥1440px` 时为 75%，且不再受固定最大宽度封顶
- 手机和平板视口 `<1024px` 时，主内容使用扣除现有安全边距后的可用宽度，不产生横向滚动
- Provider、Model、Reasoning 三个选择器不再使用原生下拉弹层，并在浅色、深色主题下保持现有视觉语言
- 没有完整 Provider 配置的用户，每次启动均会看到一次可关闭的配置向导；已配置用户不受打扰
- Chrome MV3 与 Firefox MV3 均通过功能和交互验收

## 2. 范围

### 2.1 In Scope

- 首页与聊天页共享响应式主内容宽度规则
- 首页标题、输入框、建议卡片，以及聊天消息流、底部输入框统一左右边界
- 聊天输入框内 Provider、Model、Reasoning 三个选择器改为自定义下拉
- Provider 完整性检测与首次配置引导
- 复用现有三步 `ProviderWizard`，支持新建和继续编辑残缺配置
- 无完整 Provider 时的页面内说明、配置 CTA 和禁用状态
- ProviderWizard、空配置提示及自定义下拉新增文案接入现有中英文 i18n
- Chrome MV3 与 Firefox MV3 兼容

### 2.2 Out of Scope

- 设置页及其他页面的原生下拉、复选框、单选框改造
- 首页标题、图标、建议卡片内容或视觉风格重设计
- Provider 真实连接健康检查或 API 可用性验证
- Agent、Expert、Skill、语言等其他设置项的新手引导
- 新增用户行为埋点或分析平台

## 3. 功能需求

### F1. 共享主内容容器

- 首页和聊天页必须使用同一套响应式宽度规则。
- 首页标题、首页输入框、建议卡片、聊天消息流和聊天输入框必须共享左右边界。
- 首页与聊天页切换时，不得出现由容器规则不同导致的明显横向跳变。

### F2. 桌面响应式宽度

- 可用视口 `<1024px`：主内容使用扣除现有左右安全边距后的完整可用宽度。
- 可用视口 `1024–1439px`：主内容宽度为可用视口的 90%。
- 可用视口 `≥1440px`：主内容宽度为可用视口的 75%。
- 桌面端不得再通过固定像素 `max-width` 将主内容限制到不足上述比例。
- 所有断点下均不得产生非内容自身导致的横向滚动。

### F3. 自定义聊天配置下拉

- 仅替换聊天输入框内的 Provider、Model、Reasoning 三个原生下拉。
- 触发器、菜单、悬停态、选中态、禁用态、阴影、边框和颜色必须使用现有主题变量。
- 菜单必须显示当前选中项，并在选项过多时支持内部滚动。
- 菜单必须根据可用空间向上或向下展开，并限制在当前视口内。
- 必须支持鼠标操作、方向键导航、`Enter` 选择、`Escape` 关闭和点击外部关闭。
- 关闭菜单后，焦点应返回对应触发器。
- 切换 Provider 后，继续沿用当前行为：选择该 Provider 的默认 Model，并同步 Model 的默认 Reasoning。
- 切换 Model 后，继续沿用当前行为：同步该 Model 的默认 Reasoning。
- Model 不支持 Reasoning 时，保留不可交互的 Unsupported 状态，不显示空下拉。

### F4. Provider 完整性判定

满足以下条件的 Provider 才视为完整：

- Endpoint 非空
- 至少存在一个通过现有 `ProviderWizard` 模型校验规则的 Model

API Key 不作为完整性条件，以兼容无需鉴权的本地或自建服务。

Provider 数据尚未从存储加载完成时，不得提前显示错误提示或向导。

### F5. 自动配置引导

- Provider 数据加载完成后，如果不存在完整 Provider，本次插件运行周期必须自动打开一次配置向导。
- 完全没有 Provider 时，向导从模板选择步骤开始。
- 存在残缺 Provider 时，向导必须预填第一个残缺 Provider，并直接进入最早需要补全的步骤：Endpoint 缺失进入 Connection，Model 缺失或无效进入 Models。
- 向导沿用现有 Provider 配置数据结构和保存逻辑，不创建第二套配置模型。
- Provider 目录加载失败时，用户仍可选择 OpenAI-compatible 自定义端点继续配置。

### F6. 引导关闭与再次进入

- 自动打开的向导允许关闭。
- 用户关闭后，同一运行周期内不得再次自动弹出。
- 仍无完整 Provider 时，首页必须持续显示原因说明和明确的“配置 Provider”CTA。
- CTA 必须直接重新打开 Provider 配置向导。
- 无完整 Provider 时，消息输入与建议卡片保持禁用，且禁用原因对用户可见。
- 用户下次重新打开插件时，如果仍不存在完整 Provider，向导必须再次自动打开一次。

### F7. 配置完成

- 保存完整 Provider 后，向导自动关闭。
- 新保存或补全的 Provider 自动成为当前选中 Provider。
- 自动选择该 Provider 的默认 Model；无显式默认值时选择第一个有效 Model。
- 同步选中 Model 的默认 Reasoning。
- 首页提示消失，消息输入和建议卡片立即恢复可用，无需刷新插件。

### F8. 国际化

- ProviderWizard 三个步骤中的现有硬编码英文文案必须迁移到当前 i18n 体系。
- 无 Provider 说明、配置 CTA、自定义下拉标签和状态文案必须提供中文和英文。
- 语言切换后，新增界面文案必须与应用当前语言一致。

## 4. 用户故事与验收标准

### US-1: 首次页与聊天页宽度一致

**作为** 桌面端用户
**我想要** 首页和聊天页使用一致且充分利用屏幕的内容宽度
**以便** 切换状态时界面稳定，并在宽屏上舒适阅读和输入

**验收标准：**

- [ ] 首页标题、输入框和建议卡片左右边界一致
- [ ] 聊天消息流与底部输入框左右边界一致
- [ ] 首页与聊天页的主内容左右边界遵循同一规则
- [ ] `1366px` 视口下主内容宽度为 90%
- [ ] `1920px` 和 `2560px` 视口下主内容宽度为 75%
- [ ] `<1024px` 视口下内容接近全宽且无横向滚动
- [ ] 桌面宽屏不再受固定最大宽度限制

### US-2: 使用一致风格的模型配置选择器

**作为** 聊天用户
**我想要** 在输入框内通过一致风格的下拉菜单选择 Provider、Model 和 Reasoning
**以便** 配置当前对话时不出现突兀的系统原生弹层

**验收标准：**

- [ ] 三个选择器均使用自定义菜单，不调用浏览器原生选择弹层
- [ ] 浅色和深色主题下菜单文字、背景、边框及状态均清晰可见
- [ ] 当前选中项有明确标识
- [ ] 菜单不会超出视口，并可在选项过多时滚动
- [ ] 鼠标、方向键、`Enter`、`Escape` 和点击外部关闭均正常工作
- [ ] Provider、Model、Reasoning 的联动语义与改造前一致
- [ ] 不支持 Reasoning 的 Model 显示不可交互状态

### US-3: 首次启动完成 Provider 配置

**作为** 尚未配置 Provider 的用户
**我想要** 启动后立即获得分步骤配置引导
**以便** 知道必须完成什么才能开始聊天

**验收标准：**

- [ ] Provider 加载完成且不存在完整配置时，启动后自动打开一次向导
- [ ] 无 Provider 时从模板步骤开始
- [ ] 有残缺 Provider 时预填第一个残缺配置并进入对应步骤
- [ ] 向导可关闭，关闭后同一运行周期不重复自动打开
- [ ] 关闭后首页保留说明与 CTA，输入和建议卡片保持禁用
- [ ] CTA 可直接重新打开向导
- [ ] 未完成配置时，下次启动再次自动打开一次向导
- [ ] Provider 目录请求失败不阻塞自定义端点配置

### US-4: 保存后立即开始聊天

**作为** 完成 Provider 配置的用户
**我想要** 保存后直接返回可用的聊天首页
**以便** 无需刷新或再次手动选择配置

**验收标准：**

- [ ] 保存完整 Provider 后向导自动关闭
- [ ] 保存的 Provider、默认 Model 和默认 Reasoning 自动选中
- [ ] 首页配置提示消失
- [ ] 消息输入和建议卡片立即可用
- [ ] 已有完整 Provider 的用户正常启动时不显示引导

## 5. 非功能需求

### 5.1 可访问性

- 自定义下拉必须提供正确的可访问名称、展开状态和选中状态。
- 键盘操作不得产生焦点陷阱。
- 自动向导打开后焦点进入向导，关闭后焦点回到合理的触发位置。

### 5.2 兼容性

- Chrome MV3 和 Firefox MV3 均需通过构建与交互验证。
- 不依赖仅单一浏览器支持的原生弹层行为。

### 5.3 性能

- 响应式布局和自定义下拉不得引入可感知的输入延迟。
- 首次引导除现有 Provider 目录加载外，不新增后台网络请求。

### 5.4 安全与数据

- 延用当前 Provider 存储机制，不改变 API Key 的存储和展示规则。
- 自定义下拉与引导流程不得将 Provider 凭证写入日志或界面提示。

## 6. 约束与依赖

- 基于现有 React、Tailwind CSS、主题变量和 i18n 体系实现。
- 复用 `ProviderWizard` 及现有 Provider/Model 校验语义。
- 当前代码中没有可复用的自定义 Select/Listbox 组件，具体最小实现方案在技术设计阶段确定。
- 主要相关文件：
  - `src/entrypoints/sidepanel/App.tsx`
  - `src/entrypoints/sidepanel/components/ChatView.tsx`
  - `src/entrypoints/sidepanel/components/MessageInput.tsx`
  - `src/entrypoints/sidepanel/components/ProviderWizard.tsx`
  - `src/entrypoints/sidepanel/components/SettingsPanel.tsx`
  - `src/entrypoints/sidepanel/i18n/`

## 7. 验收环境

- 视口宽度：`390px`、`768px`、`1024px`、`1366px`、`1440px`、`1920px`、`2560px`
- 状态：无 Provider、一个残缺 Provider、多个残缺 Provider、至少一个完整 Provider
- 主题：Light、Dark、System
- 语言：中文、英文
- 浏览器：Chrome MV3、Firefox MV3
