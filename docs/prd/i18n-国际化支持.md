# PRD: i18n-国际化支持

**日期:** 2026-06-29
**状态:** Draft

## 1. 概述

### 1.1 问题陈述
当前项目所有 UI 文本均为硬编码中文，无法适应英文用户的使用需求。项目已规划了 `UserPreferences.language` 字段（`zh-CN | en`），但缺乏实际的国际化基础设施和翻译实现。

### 1.2 目标用户
- 中文用户：默认体验不变
- 英文用户：首次使用自动检测语言，全界面切换为英文

### 1.3 成功指标
- 所有 UI 文本（~80+ 条）全部可切换中/英文
- 日期、时间、数字格式化跟随当前语言
- 语言切换后实时生效，无需重启扩展
- 无新增第三方 i18n 依赖，控制包体积

## 2. 功能需求

### 2.1 核心功能（MVP）
- [ ] F1: 自建 i18n 基础设施（React Context + JSON 语言包）
- [ ] F2: 首次启动自动检测浏览器语言，fallback 中文
- [ ] F3: 设置面板添加语言选择器（zh-CN / en），切换即时生效
- [ ] F4: 聊天界面全部文本 i18n（侧栏、消息气泡、输入框、状态标签等）
- [ ] F5: 设置面板全部文本 i18n（Provider/Agent/Expert/Skills Tab）
- [ ] F6: 确认对话框、错误边界、Token/浏览器状态面板 i18n
- [ ] F7: 日期/时间/数字格式化跟随语言

### 2.2 扩展功能（后续迭代）
- [ ] E1: 支持更多语言（如 ja, ko 等）
- [ ] E2: RTL 布局支持

### 2.3 非功能需求
- 性能：语言切换无感知延迟，不阻塞 UI
- 安全：语言包为纯 JSON，不含可执行逻辑
- 可维护性：新增翻译只需添加 JSON key-value，不需改组件逻辑
- 包体积：新增代码 < 5KB（gzip），不增加第三方依赖

## 3. 用户故事

### US-1: 英文用户首次使用
**作为** 英文用户
**我想要** 扩展界面自动显示英文
**以便** 无需手动配置即可使用

**验收标准：**
- [ ] 浏览器语言为 `en` 时，界面默认显示英文
- [ ] 浏览器语言为 `zh-CN` 时，界面默认显示中文
- [ ] 浏览器语言为其他语言时，fallback 到中文

### US-2: 用户手动切换语言
**作为** 任何语言用户
**我想要** 在设置面板中选择界面语言
**以便** 切换中英文界面

**验收标准：**
- [ ] 设置面板 Provider Tab 下方有语言选择器
- [ ] 选择后界面立即切换到对应语言
- [ ] 刷新/重启后语言设置保持

### US-3: 时间日期跟随语言
**作为** 英文用户
**我想要** 会话列表的时间显示格式符合英文习惯
**以便** 阅读体验一致

**验收标准：**
- [ ] 英文下时间显示 `Jun 29` 格式
- [ ] 中文下时间显示 `6月29日` 格式

## 4. 约束与假设

### 4.1 技术约束
- 必须使用自建轻量方案，不引入 i18next 等第三方库
- 语言包为纯 JSON 文件，存放在 `src/entrypoints/sidepanel/locales/` 下
- 语言切换需通过 React Context 实时传播

### 4.2 业务约束
- 首次语言检测仅在无已有偏好时执行
- 用户手动设置的语言优先级高于系统语言

### 4.3 假设
- 项目目前只有 `zh-CN` 和 `en` 两种语言需求
- 所有 UI 文本均不包含运行时动态拼接（通过模板字符串处理）

## 5. 不在范围内
- 非 UI 文本国际化（extension manifest 描述、权限提示等属于 WXT/i18n 的 chrome.i18n 范畴）
- 从右到左（RTL）排版支持
- 语音输入识别的语言切换

## 6. 附录
- `src/shared/types/storage.ts` 中 `UserPreferences.language` 已定义 `'zh-CN' | 'en'`
- `amo-metadata.json` 已声明 `supported_locales: ["en-US", "zh-CN"]`
